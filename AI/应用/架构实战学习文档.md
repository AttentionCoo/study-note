## 一、概述

LearnAgent 系统基于 LangGraph StateGraph 构建 9 个领域专家智能体。当多个专家对同一问题产生不同意见时，系统自动启动 **动态辩论-仲裁** 流程，通过多轮辩论、信任加权投票共识和仲裁裁决三层机制，确保最终输出质量。

### 1.1 为什么需要辩论机制？

单一 LLM 生成的答案存在以下风险：

| 风险 | 说明 |
|:---|:---|
| **模型偏见** | 单一模型的训练数据偏差可能导致系统性错误 |
| **幻觉放大** | 缺乏多方交叉验证，错误信息容易被当作事实输出 |
| **视角单一** | 医学问题往往需要多学科视角（影像、病理、临床、药学） |
| **证据不足** | 单一模型可能忽略关键文献证据 |

### 1.2 机制总览

系统采用 **三层协同质量保障** 架构：

```
┌─────────────────────────────────────────────────────────────────┐
│                    第一层：动态辩论（Reason Node）                │
│  多专家并行推理 → 意见冲突检测 → 多轮辩论 → 仲裁裁决             │
├─────────────────────────────────────────────────────────────────┤
│                    第二层：共识投票（Consensus Engine）           │
│  Jaccard 冲突检测 → 信任加权投票 → 跨会话信誉持久化              │
├─────────────────────────────────────────────────────────────────┤
│                    第三层：退火校验（Validate Node）              │
│  规则引擎审查 → LLM 反思审查 → 动态退火衰减 → 反射循环           │
└─────────────────────────────────────────────────────────────────┘
```

## 三、动态辩论流程（第一层）

### 3.1 动态专家编排

辩论启动前，系统首先根据 **意图类型** 和 **难度评分** 动态决定参与专家。

**配置映射**（`expert_config.yaml`）：

```yaml
dynamic_orchestration:
  enabled: true
  difficulty_thresholds:
    low: 0.3
    medium: 0.6
    high: 1.0
  intent_expert_mapping:
    resource:
      - "需求分析智能体"
      - "文档撰写智能体"
      - "题目生成智能体"
      - "质量审核智能体"
      - "学习激励智能体"
    tutor:
      - "画像对话智能体"
      - "需求分析智能体"
      - "医学影像分析智能体"
      - "质量审核智能体"
      - "学习激励智能体"
    # ... 更多意图映射
```

**仲裁智能体加入条件**：当 `difficulty_score >= 0.6` 且辩论模式开启时，自动将仲裁智能体加入参与列表。

**代码实现**（`reason_node.py` 第 200-209 行）：

```python
def _resolve_active_experts(self, state: LearningState) -> List[str]:
    active = self.expert_manager.get_experts_for_intent_and_difficulty(
        intent_type, difficulty_score
    )
    # 高难度问题自动加入仲裁智能体
    if self.debate_enabled and self.arbitrator_role not in active:
        if difficulty_score >= 0.6:
            active.append(self.arbitrator_role)
    return active
```

### 3.2 并行专家推理

所有活跃专家通过 `asyncio.gather` **并行调用** LLM，各自独立推理。

```python
tasks = []
for expert in self.experts:
    role = expert.get("role")
    if role not in active_experts:
        continue
    instruction = expert.get("instruction")
    weight = agent_weights.get(role, 1.0)
    tasks.append(self._ask_expert(role, instruction, case_info, weight))

results = await asyncio.gather(*tasks)
```

**权重衰减提示**：当某专家的 session_weight < 1.0 时（上一轮被驳回），系统会追加提示：

```python
if weight < 1.0:
    prompt += f"\n\n【注意】你在上一轮推理中部分建议被驳回，当前发言权重为{weight:.1f}，请更加谨慎地依据证据给出建议。"
```

### 3.3 辩论启动条件

满足以下条件时，辩论自动启动：

```python
if self.debate_enabled and len(expert_roles) > 1:
    debate_results = await self._run_debate(...)
```

| 条件 | 说明 |
|:---|:---|
| `debate_enabled = True` | YAML 配置中 `debate.enabled: true` |
| `len(expert_roles) > 1` | 至少 2 位专家参与（含仲裁至少 3 位） |
| `difficulty_score >= 0.6` | 高难度问题自动加入仲裁 |

### 3.4 辩论循环

辩论采用多轮迭代方式，每轮所有专家并行发言。

**辩论配置**（`expert_config.yaml`）：

```yaml
debate:
  enabled: true
  max_rounds: 1          # 当前设置为 1 轮
  arbitrator_role: "仲裁智能体"
```

**每轮辩论的 Prompt 模板**：

```
你目前参与第{round}轮辩论。以下是其他智能体的观点和之前的辩论记录：

{debate_context}

请针对与你观点不同之处提出反驳或补充，用证据支撑你的论点。如果你的观点有误，请承认并修正。
输出格式：
1. 你的立场（支持/反对/修正）
2. 论据与证据
3. 对反方观点的回应
```

**辩论上下文构建**（`_build_debate_context` 方法）：

每轮辩论前，系统构建包含以下内容的上下文：

1. **学习资料**（case_info）：学生输入 + 画像摘要 + RAG 检索证据
2. **各专家当前观点**：每轮辩论后刷新，截取前 500 字符
3. **历史辩论记录**：保留最近 6 条历史发言，截取前 300 字符

```python
def _build_debate_context(self, debate_results_map, debate_history, case_info):
    context_parts = [f"【学习资料】\n{case_info}\n"]
    context_parts.append("【各专家当前观点】")
    for role, opinion in debate_results_map.items():
        if role != self.arbitrator_role:
            context_parts.append(f"  {role}: {opinion[:500]}")
    if debate_history:
        context_parts.append("\n【历史辩论记录】")
        for record in debate_history[-6:]:
            context_parts.append(f"  第{record['round']}轮 {record['role']}: {record['content'][:300]}")
    return "\n".join(context_parts)
```

### 3.5 仲裁裁决

辩论结束后，仲裁智能体依据辩论记录和证据链做出最终裁决。

**仲裁 Prompt 模板**：

```
作为仲裁智能体，请根据以下辩论记录和证据链，对各方观点做出最终裁决。

【辩论记录】
{debate_history}

【可用证据】
{evidence}

裁决要求：
1. 评估每方论点的证据强度（强/中/弱）
2. 评估逻辑自洽性
3. 给出最终裁决结论
4. 说明裁决理由

请将裁决结果分为两部分：
### ARBITRATION ###
最终裁决结论
### REASONING ###
裁决推理过程
```

**仲裁结果注入综合汇总**：

```python
synthesis_input = expert_opinions_text
if arbitration_result:
    synthesis_input += f"\n\n【仲裁裁决】\n{arbitration_result}"
```

仲裁裁决的优先级最高，教学总监（综合汇总 LLM）在生成最终 Proposal 和 Critique 时必须优先采纳仲裁结论。

## 四、共识投票机制（第二层）

### 4.1 冲突检测

基于 **Jaccard 相似度** 判断专家意见是否存在冲突。

**算法**：计算所有专家意见对的 token 集合并比，取平均值。

```python
def detect_conflict(self, agent_advices: Dict[str, str]) -> bool:
    for i in range(len(advices)):
        for j in range(i + 1, len(advices)):
            tokens_i = set(advices[i].split())
            tokens_j = set(advices[j].split())
            intersection = tokens_i & tokens_j
            union = tokens_i | tokens_j
            jaccard = len(intersection) / len(union) if union else 0
            overlap_scores.append(jaccard)

    avg_overlap = sum(overlap_scores) / len(overlap_scores)
    return avg_overlap < self.conflict_threshold  # 默认 0.4
```

**阈值说明**：当平均 Jaccard 重叠度 < 0.4 时，判定为存在冲突。

### 4.2 信任加权投票

当检测到冲突时，启动信任加权投票机制。

**综合权重公式**：

```
combined_weight = reputation_weight × session_weight

reputation_weight = 历史正确次数 / 历史总次数（跨会话持久化）
session_weight   = 当前会话退火衰减权重（来自校验反馈）
```

**投票流程**：

```python
def resolve_conflict(self, agent_advices, session_weights=None):
    # 1. 获取信誉权重（跨会话持久化）
    reputation_weights = self.reputation_store.get_reputation_weights(roles)

    # 2. 融合会话权重（退火衰减）
    combined_weights = {}
    for role in roles:
        rep_w = reputation_weights.get(role, 0.5)
        sess_w = session_weights.get(role, 1.0) if session_weights else 1.0
        combined_weights[role] = round(rep_w * sess_w, 4)

    # 3. 归一化
    total_weight = sum(combined_weights.values())
    normalized = {r: w / total_weight for r, w in combined_weights.items()}

    # 4. 判定共识（最高权重 >= 0.6 视为达成共识）
    sorted_agents = sorted(normalized.items(), key=lambda x: x[1], reverse=True)
    top_agent = sorted_agents[0][0]
    consensus_reached = sorted_agents[0][1] >= 0.6
```

**设计原则**：不搞"少数服从多数"，而是按信誉加权。高信誉智能体的意见权重更大，低信誉智能体的异常意见被自动压制。

### 4.3 跨会话信誉持久化

信誉数据以 JSON 文件持久化到 `data/agent_reputation.json`，系统重启后不丢失。

```json
{
  "画像对话智能体": {
    "correct": 15,
    "total": 18,
    "score": 0.833,
    "last_updated": 1721200000.0
  },
  "质量审核智能体": {
    "correct": 22,
    "total": 25,
    "score": 0.88,
    "last_updated": 1721200030.0
  }
}
```

**信誉更新策略**：

| 场景 | 更新规则 |
|:---|:---|
| 校验通过 | 所有参与专家 `correct += 1` |
| 校验失败 | 权重最低的 1/3 专家 `correct -= 1`，其余 `correct += 1` |

```python
def update_reputation_from_validation(self, active_experts, validation_passed, agent_weights):
    if validation_passed:
        for role in active_experts:
            self.reputation_store.update(role, was_correct=True)
    else:
        sorted_by_weight = sorted(active_experts, key=lambda r: agent_weights.get(r, 1.0))
        low_count = max(1, len(sorted_by_weight) // 3)
        for role in sorted_by_weight[:low_count]:
            self.reputation_store.update(role, was_correct=False)
        for role in sorted_by_weight[low_count:]:
            self.reputation_store.update(role, was_correct=True)
```

## 五、动态退火校验（第三层）

### 5.1 双层校验机制

推理结果进入 `ValidateNode`，经过两层校验：

| 层 | 实现 | 说明 |
|:---|:---|:---|
| **规则引擎** | `_rule_engine_check()` | 基于 `rules_config.yaml` 中的禁忌规则做关键词匹配检查 |
| **LLM 反思** | `_llm_reflection_check()` | 质量审核 LLM 深度审查学术准确性 |

**规则引擎检查**：遍历提案文本，检测是否触发禁忌规则表中的关键词。

```yaml
# rules_config.yaml
contraindication_rules:
  资源生成:
    - 内容与知识点无关
    - 难度与学生水平不匹配
    - 缺少实例或练习
```

**LLM 反思 Prompt**：

```
作为教育质量审查专家，请校验以下学习分析与建议是否存在严重错误或安全遗漏。

【学生输入】: {case_text}
【当前综合方案 Proposal】: {proposal}

判断要求：
如果没有严重问题，请回复 "PASS"。
如果存在严重错误，请回复 "REJECT: "，并接详细的驳回理由。
```

### 5.2 动态退火策略

校验失败时，触发 **模拟退火** 机制：

**配置**（`rules_config.yaml`）：

```yaml
annealing:
  enabled: true
  weight_decay_factor: 0.5
```

**退火流程**：

```
校验失败
   │
   ├── 1. 自动分类驳回原因（5 类）
   │     ├── factual_error      事实性错误
   │     ├── logical_contradiction 逻辑矛盾
   │     ├── personalization_insufficient 个性化不足
   │     ├── medical_inaccuracy   医学专业性错误
   │     └── completeness_issue   内容不完整
   │
   ├── 2. 生成针对性修正指引
   │     例: "你的上一次输出存在事实性错误，请严格依据提供的参考证据重新推理..."
   │
   ├── 3. 权重衰减
   │     agent_weights[role] = max(agent_weights[role] × 0.5, 0.2)
   │     所有参与专家权重衰减，最低 0.2
   │
   └── 4. 反射循环
         反思次数 +1，回到 ReasonNode 重新推理
         最大反思次数: 3（validation_settings.max_reflection_count）
```

**代码实现**（`validate_node.py` 第 126-147 行）：

```python
def _apply_annealing(self, state, reason):
    category = self.validation_manager.classify_rejection(reason)
    correction_prompt = self.validation_manager.get_correction_prompt_for_category(category)

    agent_weights = dict(state.get('agent_weights', {}))
    active_experts = state.get('active_experts', [])

    for role in active_experts:
        current_weight = agent_weights.get(role, 1.0)
        if current_weight > 0.2:
            new_weight = round(current_weight * self.weight_decay_factor, 2)
            agent_weights[role] = new_weight  # 0.5 衰减因子
```

### 5.3 反射循环控制

**Graph 层路由**（`clinical_graph.py`）：

```python
graph.add_conditional_edges(
    "validate",
    self._route_validation,
    {
        "pass": "generate_report",  # 通过 → 生成报告
        "retry": "reason",          # 驳回 → 回到推理节点重试
        "fail": "generate_report"   # 达到最大反思次数 → 强制生成报告
    }
)
```

**路由逻辑**：

```python
def _route_validation(self, state):
    if state['validation_passed']:
        return "pass"
    elif state['reflection_count'] < self.max_reflection_count:
        return "retry"
    else:
        return "fail"  # 达到最大反思次数，兜底输出
```

## 六、完整数据流

### 6.1 一次完整的辩论-仲裁-校验流程

```
Step 1: 意图识别 + 需求分析 + RAG 检索
   │
Step 2: ReasonNode 动态编排活跃专家
   │  intent_type=resource, difficulty_score=0.8
   │  → 需求分析 + 文档撰写 + 题目生成 + 质量审核 + 学习激励 + 仲裁
   │
Step 3: 并行专家推理（6 位专家同时调用 LLM）
   │  收集各专家建议 → expert_advices
   │
Step 4: 辩论（1 轮）
   │  构建辩论上下文 → 各专家发言 → 辩论记录
   │
Step 5: 仲裁裁决
   │  仲裁智能体评估证据强度 → 最终裁决
   │
Step 6: 共识投票（共享记忆层）
   │  Jaccard 冲突检测 → 信任加权投票 → 更新信誉
   │
Step 7: 意见综合（教学总监）
   │  专家意见 + 仲裁裁决 → 综合 Proposal + Critique
   │
Step 8: ValidateNode 双层校验
   │  规则引擎 → LLM 反思
   │
   ├── PASS → GenerateReport
   │
   └── REJECT → 退火衰减权重 → 回到 Step 3（最多 3 次）
```

### 6.2 状态流转示意

```
LearningState 关键字段流转：

intent_type ──────── intent_node ────► 进入各节点
difficulty_score ─── analysis_node ──► 决定仲裁是否加入
evidence ─────────── retrieve_node ──► 供辩论和仲裁使用
agent_weights ────── validate_node ──► 退火衰减（0.5 因子）
debate_history ───── reason_node ────► 累计辩论记录
arbitration_result ─ reason_node ────► 注入综合汇总
consensus_result ─── reason_node ────► 共识投票结果
reflection_count ─── validate_node ──► 反射次数控制
validation_feedback─ validate_node ──► 驳回原因 + 修正指引
rejection_categories validate_node ──► 驳回分类累计
```

| 参数 | 位置 | 默认值 | 说明 |
|:---|:---|:---|:---|
| `conflict_threshold` | ConsensusEngine | `0.4` | Jaccard 冲突检测阈值 |
| `min_agreement_ratio` | ConsensusEngine | `0.6` | 最低共识比例 |

### 7.3 退火校验参数

| 参数 | 位置 | 默认值 | 说明 |
|:---|:---|:---|:---|
| `annealing.enabled` | rules_config.yaml | `true` | 是否启用退火 |
| `weight_decay_factor` | rules_config.yaml | `0.5` | 权重衰减因子 |
| `max_reflection_count` | rules_config.yaml | `1` | 最大反思次数（建议 1-3） |
| `enable_rule_engine` | rules_config.yaml | `true` | 是否启用规则引擎 |
| `enable_llm_reflection` | rules_config.yaml | `true` | 是否启用 LLM 反思 |

### 7.4 性能调优建议

| 场景 | 建议 |
|:---|:---|
| **追求速度** | 关闭辩论（`debate.enabled: false`），关闭 LLM 反思（`enable_llm_reflection: false`） |
| **追求质量** | 开启辩论 + 2 轮辩论 + 开启 LLM 反思 + `max_reflection_count: 3` |
| **平衡模式** | 辩论 1 轮 + LLM 反思 + `max_reflection_count: 1`（当前默认配置） |
| **降低 Token 消耗** | 减少参与专家数（通过 `intent_expert_mapping` 精简），降低辩论轮数 |
