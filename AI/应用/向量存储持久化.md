## 第一章：为什么需要持久化

### 1.1 核心问题

在 RAG（检索增强生成）系统中，向量库的作用是将文档转换为向量并存储，以便后续通过语义检索召回相关内容。如果不做持久化，每次系统重启都需要重新解析文档、重新计算 Embedding、重新构建向量索引——这会带来两个严重问题：

1. **启动时间过长**：大型文档集（如数百页 PDF）的 Embedding 计算可能需要数分钟甚至数十分钟。
2. **资源浪费**：每次重启都重复相同的计算，浪费 CPU 和 API 调用额度。

### 1.2 持久化的价值

| 维度 | 无持久化 | 有持久化 |
|------|---------|---------|
| 启动速度 | 每次数分钟 | 秒级加载 |
| 知识延续性 | 重启即丢失 | 跨会话保留 |
| 共享记忆 | 无法跨 Agent 复用 | 多 Agent 共同沉淀 |
| 运维成本 | 高 | 低 |

## 第二章：整体架构设计

本项目采用 **双库分离** 的持久化策略，将主知识库与共享记忆库分开管理：

```
┌──────────────────────────────────────────────────────────────────┐
│                        向量库持久化体系                            │
├────────────────────────────┬─────────────────────────────────────┤
│     主知识库（RAG）          │       共享记忆库（SharedMemory）      │
│   chroma_db_unified/       │    chroma_db_shared_memory/         │
│   ─────────────────        │    ────────────────────────         │
│   来源：PDF 文档解析          │    来源：多 Agent 对话中提取的知识      │
│   构建：build_or_load()    │    构建：SharedMemoryStore.store()   │
│   用途：医学知识检索增强       │    用途：跨会话知识沉淀与复用           │
│   触发：系统启动 / 手动脚本    │    触发：每次 Agent 产生有价值输出       │
├────────────────────────────┴─────────────────────────────────────┤
│                  底层：ChromaDB（PersistentClient）                │
│                Embedding：XfyunEmbeddings（1024d BGE）             │
│                 过滤：MetaMemoryFilter（熵值阈值 0.85）              │
│                 共识：ConsensusEngine（冲突解决）                    │
│                 信誉：AgentReputationStore（JSON 持久化）            │
└──────────────────────────────────────────────────────────────────┘
```

**设计理念**：

- **主知识库**：存放从 PDF 文档中提取的静态知识，内容相对稳定，在文档更新时重建。
- **共享记忆库**：存放多 Agent 协作过程中动态产生的知识洞察，持续积累，用熵值过滤保证质量。

## 第三章：核心组件详解

### 3.1 XfyunEmbeddings — 双通道 Embedding 引擎

XfyunEmbeddings 是整个向量体系的发动机，负责将文本转换为向量。

**调用链：**

```
embed_query / embed_documents
  → 讯飞 API（HMAC-SHA256 签名鉴权）
  → 调用失败？→ _xfyun_dead = True
  → _get_fallback_embeddings()
  → 本地 BGE-large-zh-v1.5（1024d，CPU 推理）
```

**关键设计机制：**

| 机制 | 说明 |
|------|------|
| 签名鉴权 | `_embed_once()` 内使用 HMAC-SHA256，携带 `X-App-Id` 等 4 个请求头 |
| QPS 节流 | `_throttle()` 类方法，`_last_request_time` 类变量，请求间隔 ≥ 0.7s |
| 错误分级 | `_FATAL_ERROR_CODES` 字典，错误码 11200/11201/10001 等标记为不可重试 |
| 批量降级 | 单条失败时整批统一降级，避免同一批次中出现维度混用 |
| 重试策略 | 指数退避 `1.5 × (attempt + 1)` 秒，最多重试 4 次 |

**BGE 模型类级缓存（核心优化）：**

```python
# 修复前：实例变量，每次 XfyunEmbeddings() 实例化都重新加载 1.3GB 模型
self._fallback_embeddings = None

# 修复后：类变量，全进程共享一份
_fallback_embeddings_cache = None   # 类级别缓存
_fallback_embeddings_failed = False # 失败标记，避免重复重试

@classmethod
def _get_fallback_embeddings(cls):
    if cls._fallback_embeddings_cache is not None:
        return cls._fallback_embeddings_cache  # 缓存命中，瞬时返回
    if cls._fallback_embeddings_failed:
        return None                            # 已知失败，不再重试
    # 首次加载，耗时约 3 分钟
    cls._fallback_embeddings_cache = HuggingFaceBgeEmbeddings(...)
    return cls._fallback_embeddings_cache
```

**效果**：第二次及之后实例化 `XfyunEmbeddings()` 从 185 秒降至 0 秒。

### 3.2 SharedMemoryStore — 共享记忆存储

多 Agent 共享知识记忆库，实现跨会话的知识沉淀。

**核心能力：**

- `store()`：将 Agent 的知识洞察存入向量库，附带元数据（来源 Agent、时间戳、置信度）。
- `search()`：基于语义相似度检索历史记忆，辅助当前决策。
- 基于 ChromaDB PersistentClient，自动持久化到磁盘。

---

### 3.3 MetaMemoryFilter — 元记忆过滤器

并非所有 Agent 输出都值得存入记忆。MetaMemoryFilter 通过 **信息熵** 评估内容价值：

```python
class MetaMemoryFilter:
    def __init__(self, entropy_threshold=0.85):
        self.threshold = entropy_threshold

    def filter(self, items):
        """高价值保留（熵值 < 0.85），低价值丢弃（熵值 ≥ 0.85）"""
        return [item for item in items
                if self._calculate_entropy(item.content) < self.threshold]
```

**设计原理**：熵值越高，信息越"泛"（如"你好"、"收到"等通用回复），这些内容缺乏存储价值；熵值越低，信息越具体，具有长期参考意义。

### 3.4 ConsensusEngine — 共识引擎

当多个 Agent 对同一问题给出不同建议时，通过投票机制解决冲突。

```python
advices = {
    "agent_a": "建议从 Willis 环解剖开始...",
    "agent_b": "建议从 Willis 环解剖开始...",  # 与 a 一致
    "agent_c": "跳过解剖，直接学指南...",       # 与 a/b 冲突
}

result = ConsensusEngine().resolve_conflict(advices)
# → winning_agents: ["agent_a", "agent_b"]  （多数派）
# → reached: False（未达 100% 共识）
# → 阈值: 0.4（40% 以上即可形成多数派决议）
```

### 3.5 AgentReputationStore — Agent 信誉存储

追踪每个 Agent 的信誉分数，以 JSON 文件持久化。

```python
store = AgentReputationStore(config={"reputation_file": "agent_reputation.json"})
store.update("agent_a", was_correct=True)   # 正确 +1
store.update("agent_a", was_correct=False)  # 错误 +1
score = store.get_score("agent_a")          # 0.5 (1/2)
```

**持久化格式：**

```json
{
  "agent_a": {"correct": 10, "incorrect": 5},
  "agent_b": {"correct": 8,  "incorrect": 2}
}
```

## 第四章：维度一致性保证

### 4.1 核心原则

**所有向量库必须使用同一份 Embedding 模型（1024d BGE），否则检索结果不可比。**

```
XfyunEmbeddings（1024d BGE）
  ├── _ChromaEmbeddingFunction 包装
  │     ├── SharedMemoryStore → chroma_db_shared_memory
  │     └── build_or_load_vectorstore → chroma_db_unified
  └── 直接调用
        ├── embed_query() → 检索用
        └── embed_documents() → 入库用
```

### 4.2 配置开关

在 `.env` 中设置 `XFYUN_EMBEDDING_ENABLED=false`，跳过讯飞云端 API，全程使用本地 BGE 模型，从根源上保证维度统一。

## 第七章：常见问题与解决方案

| 问题 | 根因分析 | 解决方案 |
|------|----------|----------|
| ChromaDB 初始化失败 | `__call__(texts)` 参数名与 ChromaDB 内部调用不匹配 | 改为 `__call__(input)` 以兼容 ChromaDB 接口 |
| BGE 模型重复加载 | 实例变量无法跨实例共享，导致每次新建都重新加载 | 改为类级别 `_fallback_embeddings_cache` |
| 向量维度不一致 | 旧库用 ONNX-384d，新库用 BGE-1024d，维度冲突 | 删除旧库目录，用统一维度重建 |
| 运行时静默"假死" | BGE 首次下载约 3 分钟，期间无日志输出 | main.py 启动时主动调用 `preload_fallback()` |
| Windows 清理报 PermissionError | ChromaDB 文件锁未释放 | 重启进程后清理，或忽略该错误 |
