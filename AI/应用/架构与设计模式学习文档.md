# AI 应用架构权威指南：设计模式与编排流程深度解析

在构建复杂的 AI 应用（如深度垂直领域的智能助手、自动化决策引擎）时，系统架构通常需要分层设计。为了让 AI 系统稳定、可控且高效地运行，开发者必须清晰区分**设计模式（Design Patterns）**与**编排流程（Orchestration Flows）**。

- **设计模式是“架构蓝图”**：解决“用什么思路来应对这类复杂问题”。
- **编排流程是“流水线工程”**：解决“在代码层面如何调度任务和传递状态”。

---

## 一、 核心概念：模式与流程的区别

| 维度 | 设计模式 (Design Pattern) | 编排流程 (Orchestration Flow) |
| :--- | :--- | :--- |
| **本质** | 解决特定问题的通用交互范式与策略 | 代码实现层的执行步骤与状态流转 |
| **解决问题** | “AI 的大脑应该如何思考、决策和使用工具？” | “下一步具体调用哪个 API？节点间状态如何传递？” |
| **具象比喻** | 建筑设计的风格模板（如：现代极简风格） | 施工队的详细作业工序表和排期 |
| **稳定性** | 极高，属于底层思想，跨框架/跨语言通用 | 易变，随业务逻辑增删和提示词微调而频繁修改 |
| **典型代表** | Tool Use、Reflection、RAG、Plan-and-Solve | StateGraph、Sequential Chain、If-Else 路由 |

设计模式提供了思想的“骨架”，它不关心实现细节；编排流程则是把骨架填充为可运行的血肉。在构建系统时，先根据业务需求选定合适的设计模式（或模式组合），再以编排框架将模式落地为具体的执行图。

---

## 二、 业界主流的 5 大核心设计模式

设计模式决定了智能体（Agent）的智商上限和能力边界。以下是目前构建复杂 AI 系统最核心的几种架构模式。我们将逐一剖析其原理、核心组件与角色，并给出场景示例。

### 1. 工具调用模式 (Tool Use / ReAct)

**原理**  
让语言模型突破文本生成的局限，使其能够与外部世界交互。模型基于输入的问题，通过 `Thought（思考） → Action（行动） → Observation（观察反馈）` 的循环，自主决定是否调用工具（如数据库查询、计算器、API）并解析工具返回的结果。

**核心组件与角色**  
- **Agent 主体**：一个能生成“思考-行动”序列的 LLM 实例，通常被赋予一个系统提示，要求其输出结构化的行动指令（如 JSON 或特定标记）。  
- **工具集（Toolbox）**：一组可被调用的外部函数或 API，每个工具需要明确定义名称、描述、参数 schema，以便模型准确理解工具的用途和调用方式。  
- **执行器（Executor）**：负责解析模型的行动指令，实际调用对应工具，并将返回结果转化为可读的“观察”文本。  
- **记忆状态（Memory）**：保存当前会话中已经执行的所有“思考-行动-观察”步骤，使 Agent 具备多步推理上下文。  
- **终止判定逻辑**：当模型输出最终答案而不再需要调用工具时，流程结束。

**医疗应用场景示例**  
当用户询问某药物的相互作用时，模型不依赖内部模糊记忆，而是决定调用“临床指南检索 API”，观察返回的准确医学文献后，再生成严谨的用药建议。整个过程受工具执行器严格控制，避免模型凭空编造药理数据。

---

### 2. 自我反思模式 (Self-Reflection)

**原理**  
不信任模型的单次输出（Zero-shot），通过引入评估-修正机制来提升质量。设定一个 Actor（生成者）和一个 Evaluator（评估者）。Actor 生成初步答案，Evaluator 负责审查错误（如检查是否产生幻觉、是否遗漏条件）并给出改进建议（Critiques），Actor 接收建议后再次生成，直到满足阈值。

**核心组件与角色**  
- **生成器（Actor/Generator）**：负责生成内容的主体 LLM，接收任务描述和反馈历史后产出答案。  
- **评估器（Evaluator/Critic）**：可以是另一个更强大的 LLM，也可以是规则引擎、外部检查器。它对生成内容进行多维度打分（事实准确性、逻辑一致性、格式合规性等），输出评分和改进建议。  
- **反馈历史（Critique Memory）**：记录每次评估给出的批评和对应的修改建议，供生成器在下一次迭代时参考，避免重复同样错误。  
- **短期轨迹记忆（Episodic Memory）**：记录当前会话的思维链和试错过程，让模型能“记住”刚才哪里犯了错。  
- **长期经验记忆（Semantic Experience）**：跨会话总结的通用修正法则，比如“遇到用药相关问题时，务必要核对患者过敏史”。  
- **终止控制器（Stopping Gate）**：根据评分阈值、迭代次数或特定退出信号决定是否停止反思循环。

**扩展说明**  
自我反思模式在医疗、法律等高严谨性场景中尤为重要。它可以有效抑制幻觉，但必须设计好终止条件，否则容易陷入无限修改的死循环。实践中常结合“最大重试次数”和“外部事实校验 API”作为硬退出机制。

### 3. 规划与执行模式 (Planning / Plan-and-Solve)

**原理**  
处理需要多步推理的复杂宏大任务。AI 首先充当“规划师”，将复杂指令拆解为一系列连贯的子任务列表。随后，AI 按顺序或并行执行这些子任务，并将中间结果进行合并与推导。这种模式将“想清楚再干活”的策略制度化。

**核心组件与角色**  
- **规划器（Planner）**：一个擅长任务分解的 LLM，它接收总目标并输出结构化的步骤列表（Plan），每个步骤可能有前置依赖。  
- **步骤执行器（Step Executor）**：依次读取计划中的每一步，调用必要的工具或子模型执行，记录每一步的输出结果。  
- **状态管理（State Context）**：保存已执行步骤的结果、当前进度和中间数据，供后续步骤使用。  
- **重规划机制（Replanner）**：当某一步执行失败或结果不符合预期时，可触发重规划，动态调整剩余步骤。  
- **聚合器（Aggregator）**：在所有步骤完成后，将所有结果合并成最终答案，可能进行二次润色或总结。

**场景示例**  
用户要求“总结这位患者的过去三个月的病情变化并输出转诊报告”。模型先拆解任务：1. 检索三个月内的门诊记录；2. 提取关键体征指标变化趋势；3. 按照标准格式生成转诊说明；4. 校对报告中的机构名称和日期。每个步骤由执行器顺序完成，最终聚合器输出格式整洁的转诊单。

### 4. 多智能体协作模式 (Multi-Agent Collaboration)

**原理**  
分工明确，让“专业的人做专业的事”，通过不同角色的 Agent 协同完成复杂业务。在一个系统内定义多个具备特定系统提示词（Persona）和专属工具的 Agent。通过消息传递机制，让它们相互对话、质询或接力工作。

**核心组件与角色**  
- **多智能体集群**：每个 Agent 被赋予固定的角色（如研究员、审核员、报告写手）、个性提示词和专属工具箱。  
- **通信总线（Message Bus）**：定义 Agent 之间如何发送消息，可以是点对点对话、群聊广播或有管理员控制的轮流发言。  
- **对话管理器（Conversation Manager）**：决定发言顺序、何时将对话焦点从 A 移交给 B，可能实现为“群聊管理员（Group Chat Manager）”。  
- **共享黑board（Shared Workspace）**：所有 Agent 可见的上下文存储区，用于传递中间结果、数据表格、草稿等，避免信息极度膨胀。  
- **冲突与裁决机制**：当多个 Agent 意见不合时（如一个建议用药，另一个指出过敏风险），需要仲裁逻辑决定最终输出或触发反思链。

**场景示例**  
构建一个医疗辅诊系统。Agent A（分诊员）负责意图识别和信息收集；Agent B（临床决策大脑）负责基于 RAG 提供诊断建议；Agent C（合规审核员）负责审查 B 的输出是否符合医疗安全规范。消息流为：A 收集完主诉和检验数据后传递给 B，B 生成疑似诊断和用药建议，C 审核并可能打回修改，最终经 A 返回给用户。

### 5. 检索增强生成 (RAG)

**原理**  
解决模型知识滞后和私有数据不可见的问题。通过将外部文档向量化，在用户提问时优先检索出相关文本块（Chunks），将检索结果作为上下文注入给大模型，限制其只能基于给定上下文作答。

**核心组件与角色**  
- **文档加载与解析器（Loader & Parser）**：将各种格式（PDF、网页、数据库）的原始文档转换为纯文本。  
- **分块器（Chunker）**：按语义边界将长文档切分为合适大小的片段（Chunks），控制嵌入质量和检索粒度。  
- **嵌入模型（Embedding Model）**：将文本块转换为高维向量，生成向量空间的语义表征。  
- **向量数据库（Vector Store）**：存储向量及其对应的元数据，提供高效的相似度检索。  
- **检索器（Retriever）**：接收用户查询向量，从向量库中搜索最相关的 top‑k 个文本块。  
- **生成器（Generator）**：即 LLM，它以拼接好检索结果的完整 Prompt 作为输入，生成基于证据的最终回答。  
- **可选的重排序器（Re‑ranker）**：对初步检索结果进行更精细的相关性打分与重排，提升输入生成器的上下文质量。

**扩展说明**  
RAG 常与其他模式组合使用：例如在多智能体协作中，某个 Agent 的专属工具就是 RAG 管线；或者在反思模式中，评估器可依据检索到的文本核实生成内容的准确性。

## 三、 典型的编排流程与路由机制

有了上述“设计蓝图”，我们需要通过编排流程在底层系统（如 FastAPI 后端）中将它们跑起来。这里介绍几种常见的代码流转拓扑结构及其核心组件。

### 1. 顺序链状流 (Sequential Pipeline)

- **机制**：`节点A → 节点B → 节点C`。  
- **核心组件**：顺序节点列表、数据传递管道。每个节点从上游接收输入，处理后将输出原样或转换后传给下游。  
- **应用**：经典的单路 RAG 流程。用户输入 → 文本嵌入（Embedding） → 向量库检索 → 拼接 Prompt → LLM 生成。流程单向、线性，不存在分支。

### 2. 路由分发图 (Router / Classifier)

- **机制**：在入口处设置一个轻量级的分类器节点，基于用户意图决定走哪一条分支执行。  
- **核心组件**：  
  - **意图分类器**：可以是一个微调的小模型或借助 LLM 的 Function Calling 实现。  
  - **条件边（Conditional Edge）**：根据分类结果将流程导向不同的后续节点。  
  - **分支处理节点**：每个分支可能是一条独立的子流程。  
- **应用**：客服/问诊网关。若是日常闲聊，直接路由到轻量级模型生成；若是查阅特定病历，路由到带有患者数据库读取权限的深层处理节点。

### 3. 评估-优化循环图 (Evaluator-Optimizer Loop)

- **机制**：一个包含回路的状态机（State Machine）。通常通过图结构中的“条件边”来实现。  
- **核心组件**：  
  - **生成节点（Generator Node）**：输出待评估的内容。  
  - **评估节点（Evaluator Node）**：对内容进行评分并产生反馈。  
  - **状态变量（score, iteration_count）**：存储当前评分和重试次数。  
  - **条件路由逻辑**：若分数 < 阈值且未达最大次数，则返回生成节点；否则走向结束。  
- **应用**：**Self-Reflection 设计模式**的具体编排实现。执行流进入评估节点后，如果 `score < 0.8`，条件边将状态路由回生成节点重新处理；如果 `score >= 0.8` 且没有发生外部系统异常，条件边路由到终点（END）。

### 4. 并行执行与聚合 (Parallel Execution & Fan-in)

- **机制**：同时触发多个处理节点，等待所有节点返回后，由一个聚合节点进行总结。  
- **核心组件**：  
  - **扇出（Fan-out）**：从单个节点分发任务到多个并行分支。  
  - **并行执行器**：利用异步调用同时运行多个独立任务。  
  - **聚合/合并节点（Fan-in / Reducer）**：收集所有分支结果，合并或择优后输出。  
- **应用**：在执行多路召回（混合检索）或需要让多个模型针对同一问题提供不同视角的分析报告时，极大提升系统响应速度。

## 四、 主流框架的设计哲学对比

理解了模式与流程，在技术选型时就能有的放矢。目前主流 AI 框架在两者上的侧重点截然不同：

1. **LangChain / LangGraph**：**重编排、显式状态图**。它的核心是 `StateGraph`，提供了极其强大的节点（Node）和边（Edge）定义能力。它允许你通过代码极细粒度地控制流转逻辑，非常适合落地复杂的业务级容错、断点重试和并发控制。适合需要精确控制每一步行为的工程团队。

2. **DSPy**：**重模式、隐式编排**。你只需声明式地定义系统包含哪些“模块”（例如声明这里需要一个 RAG 模块、那里需要一个 ReAct 模块），DSPy 的编译器会在底层自动优化模型内部调用的流转和提示词，大幅减少手动干预流程的工作。适合希望快速实验不同模式组合、且信任编译器优化能力的研究者。

3. **LlamaIndex**：专注 **RAG 模式的深度编排**。在文档解析、分块（Chunking）、多级索引（Indexing）和高级检索路由方面，提供了开箱即用的深度抽象。适合以海量非结构化数据为基础、需要构建复杂检索管线的应用。

4. **AutoGen**：专注 **多智能体（Multi-Agent）模式的对话编排**。通过定义 Agent 间的对话机制（如群聊管理员、按顺序发言等），将复杂的流程编排转化为多角色的自然语言对话交互。适合需要多角色博弈、协商、接力处理的协作式 AI 系统。

## 五、 实战指南：在系统中如何落地反思循环？

结合上述理论，如果我们要在一个严谨的系统中实现“安全的反思循环”，核心在于**控制终止条件**。无论编排框架多么先进，如果不加限制，模型很可能陷入无限纠错的死循环。

### 终止条件设计的三个原则：

1. **质量满足阈值 (Quality Satisfied)**：正常跳出。评估器对当前结果在准确性、一致性等方面打分达标（例如通过了某一项合规校验 API 的测试），流程结束。阈值的设定需经过生产数据验证，平衡质量与耗时。

2. **触发最大步数兜底 (Max Iterations)**：安全跳出。必须在系统的全局状态（State）中维护一个 `iteration_count` 变量。设定 `max_iterations = 3`，一旦超过直接截断并返回当前最优解或转交人工介入。这是防止死循环的最后防线。

3. **主动宣告结束 (Explicit Termination)**：逻辑跳出。模型由于确认自身缺乏必要信息（例如发现患者指标不足以支持推断），输出特定标识符（如 `CANNOT_ANSWER`），触发流程提前结束，避免产生强行生成的幻觉。这要求评估器能够识别“信息不足”与“答案错误”的区别。

### 额外稳健性建议

- **外部校验钩子**：在关键领域（如药物剂量、手术禁忌），插入规则引擎或确定性 API 验证，一旦命中硬性冲突，无论 reflection score 多高都直接否决。  
- **可观测性埋点**：在生成节点、评估节点和条件边上打入详细日志，记录每次循环的输入输出、评分和决策缘由，便于事后复盘和提示词调优。  
- **降级策略**：当反思循环被强制终止时，应根据业务场景给出优雅的降级输出，例如“抱歉，目前无法给出确切答案，建议咨询主治医师”。

通过将设计模式的思想深植于系统顶层，再用灵活的编排流程加以实现，并辅以周全的安全控制机制，开发者才能构建出既聪明又可靠的工业级 AI 应用。

## 六、多智能体角色扮演模式

> 详见子文档：[多智能体系统.md](file:///D:/Note/AI/应用/多智能体系统.md)

### 6.1 核心概念

**多智能体角色扮演模式**，是指为不同智能体赋予特定的"角色身份"（专业领域、职责、目标），让它们通过拟人化的对话与协作共同完成复杂任务。本质是将思维链从单一模型内部**外化**为多个角色间的社交互动。

### 6.2 与传统单智能体的区别

| 单智能体 | 多智能体角色扮演 |
|:---|:---|
| 一个人想所有角度，容易思维固化 | 多个专家从不同视角审视，减少盲点 |
| 缺乏外部批判，容易产生幻觉 | 相互验证指正，显著提高事实准确性 |
| 任务流程线性，仅靠提示词控制 | 动态协作，可在讨论中自然涌现新方案 |
| 决策过程不透明 | 对话流即是可解释的"思考过程" |

### 6.3 三层系统架构

```
┌─────────────────────────────────────────────────┐
│  智能体层（角色定义与认知核心）                    │
│  角色设定 + LLM 引擎 + 短期/长期记忆               │
├─────────────────────────────────────────────────┤
│  交互层（对话、路由与协调）                        │
│  对话协议 + 消息路由（回合制/按需发言/主持人仲裁）    │
│  + 状态管理（共享黑板）                           │
├─────────────────────────────────────────────────┤
│  环境层（共享空间与工具）                          │
│  共享工作区 + 工具接口（代码执行、API 调用等）       │
└─────────────────────────────────────────────────┘
```

### 6.4 典型框架

- **AutoGen**：对话驱动架构，预定义对话模式组织协作
- **ChatDev**：瀑布式流程架构，模拟软件公司分阶段开发

## 七、推理与规划范式

> 详见子文档：[推理与规划范式.md](file:///D:/Note/AI/应用/推理与规划范式.md) 和 [思维链CoT深度解析.md](file:///D:/Note/AI/应用/思维链CoT深度解析.md)

### 7.1 四种 Planner 方案对比

| 方案 | 思考方式 | 外部交互 | 灵活性 | 成本 | 典型场景 |
|:---|:---|:---|:---|:---|:---|
| **CoT** | 线性推理，不行动 | 否 | 低 | 低 | 数学题、逻辑推理 |
| **ReAct** | 交替思考与行动 | 是 | 高 | 高 | 搜索引擎问答、API 操作 |
| **Plan-and-Solve** | 先完整计划再逐步执行 | 是 | 中低 | 中 | 结构化自动化流程 |
| **ToT** | 树状探索与回溯 | 可结合 | 非常高 | 极高 | 复杂谜题、策略游戏 |

### 7.2 CoT 深度解析

**核心原理**：将单步直接预测转化为多步条件预测，增加推理路径长度为模型提供更多"计算空间"。

**关键变体**：

| 变体 | 原理 | 适用场景 |
|:---|:---|:---|
| Zero-shot CoT | 提示末尾加 "让我们一步步思考" | 通用任务，快速验证 |
| Few-shot CoT | 提供 2-3 个带完整推理步骤的示例 | 需要特定格式输出的任务 |
| Auto-CoT | 自动聚类 + 采样 + 生成示例链 | 大规模任务，避免人工挑选偏差 |
| Self-Consistency | 并行采样多条推理路径，多数投票 | 数学推理、逻辑题（提升 5-10%） |

**CoT → ReAct 演化**：在推理过程中引导模型输出可执行的"动作标记"，由外部系统执行反馈，CoT 就演化成了 ReAct。

### 7.3 选型建议

| 任务特征 | 推荐方案 |
|:---|:---|
| 简单，不依赖外部信息 | CoT |
| 需要联网或操作工具 | ReAct |
| 流程固定，步骤明确 | Plan-and-Solve |
| 非常难，需要深度探索 | ToT（算力允许时） |

实际应用中常混合使用：先用 Plan-and-Solve 生成高层计划，每个步骤内部用 ReAct 执行。

## 八、LangGraph 框架核心

> 详见子文档：[LangGraph框架.md](file:///D:/Note/AI/应用/LangGraph框架.md)

### 8.1 定位

```
LangChain = 单链路调用
LangGraph = 多节点状态机（State Machine）+ Agent 工作流编排
```

LangGraph 特别适合：多 Agent 系统、RAG 系统、知识图谱问答系统。

### 8.2 四大核心概念

```
State（状态） → Node（节点） → Edge（边） → Graph（图）
```

### 8.3 关键机制速查

| 机制 | 说明 |
|:---|:---|
| **State 定义** | 使用 `TypedDict`，所有节点共享读写 |
| **状态更新** | 累积合并，节点只返回需要修改的字段 |
| **条件路由** | `add_conditional_edges()` 根据状态动态决定下一步 |
| **编译执行** | `graph.compile()` 后通过 `app.invoke(state)` 执行 |
| **流式输出** | 支持 `stream()` 实时获取中间状态 |
| **Memory** | 内置 `MemorySaver` 检查点，支持断点续传和时间旅行 |

### 8.4 本项目实战映射

LearnAgent 的 `clinical_graph.py` 完整实现了 LangGraph 工作流：

```
IntentNode → AnalysisNode → RetrieveNode → ReasonNode → ValidateNode
                                                    ↕（辩论循环）
                                         ValidateNode → ReportNode
                                                    ↕（反思循环，最多 3 次）
```

## 九、工程实践精要

> 详见子文档：[工程实践.md](file:///D:/Note/AI/应用/工程实践.md)

### 9.1 工作流四要素

| 要素 | 含义 | 大模型项目举例 |
|:---|:---|:---|
| **节点 (Steps)** | 具体要执行的任务 | 数据清洗、模型评测、部署上线 |
| **流转 (Flow)** | 步骤间的先后顺序与依赖 | 必须先完成预训练，才能进行 SFT |
| **输入/输出 (I/O)** | 上一步产出如何成为下一步原料 | 清洗后的语料 → 训练框架的数据接口 |
| **角色/工具 (Actors/Tools)** | 由人完成还是系统自动调度 | 工程师搭建集群 / CI/CD 管道 |

### 9.2 Skill 与 MCP：流程 + 工具的协同

| 维度 | MCP | Skill |
|:---|:---|:---|
| **本质** | 打通外部工具和数据的标准化"万能接口" | 将专业知识、流程封装成可遵循的"SOP 操作手册" |
| **核心问题** | "我能调用什么能力？" | "我应该如何规范地执行？" |
| **知识类型** | 工具性知识（Tool Knowledge） | 流程性知识（Procedural Knowledge） |
| **载体形式** | MCP 服务器（JSON-RPC 协议） | 基于 Markdown 的文件夹，渐进式披露 |
| **典型场景** | 查天气、调 API、操作数据库 | 写周报、代码审查、按品牌规范输出 |

**抽象公式**：`通用大模型（大脑） + MCP（手脚与工具） + Skills（专业经验与流程） = 高质量、可落地的 AI 应用`

### 9.3 反思循环落地的终止条件设计

1. **质量满足阈值**：评估器打分达标，正常跳出
2. **触发最大步数兜底**：`iteration_count >= max_iterations`（建议 3），安全跳出
3. **主动宣告结束**：模型输出 `CANNOT_ANSWER`，逻辑跳出

## 十、架构实战：多智能体动态辩论-仲裁机制

> 详见子文档：[架构实战学习文档.md](file:///D:/Note/AI/应用/架构实战学习文档.md)

### 10.1 三层协同质量保障

```
┌─────────────────────────────────────────────────────────────────┐
│  第一层：动态辩论（Reason Node）                                  │
│  多专家并行推理 → 意见冲突检测 → 多轮辩论 → 仲裁裁决              │
├─────────────────────────────────────────────────────────────────┤
│  第二层：共识投票（Consensus Engine）                             │
│  Jaccard 冲突检测 → 信任加权投票 → 跨会话信誉持久化               │
├─────────────────────────────────────────────────────────────────┤
│  第三层：退火校验（Validate Node）                                │
│  规则引擎审查 → LLM 反思审查 → 动态退火衰减 → 反射循环            │
└─────────────────────────────────────────────────────────────────┘
```

### 10.2 核心机制速览

| 机制 | 关键设计 | 技术要点 |
|:---|:---|:---|
| **动态专家编排** | 意图类型 + 难度评分 → 动态选择专家 | `difficulty_score >= 0.6` 自动加入仲裁 |
| **并行推理** | `asyncio.gather` 并行调用所有专家 LLM | 权重衰减提示（`session_weight < 1.0` 时追加警告） |
| **辩论循环** | 多轮并行发言，每轮构建上下文 | 上下文含：学习资料 + 各专家观点 + 历史辩论记录（最近 6 条） |
| **仲裁裁决** | 仲裁智能体评估证据强度 → 最终裁决 | 裁决结果分为 `### ARBITRATION ###` 和 `### REASONING ###` 两部分 |
| **共识投票** | Jaccard 相似度冲突检测 + 信任加权投票 | `combined_weight = reputation_weight × session_weight` |
| **退火校验** | 双层校验（规则引擎 + LLM 反思）+ 模拟退火 | 权重衰减因子 0.5，最低 0.2；最大反思 3 次 |
| **信誉持久化** | JSON 文件跨会话保存 | 校验通过全员 +1，失败则权重最低 1/3 专家 -1 |

### 10.3 退火重试循环

```
ReasonNode → ValidateNode
     ▲            │
     │            ├── PASS → ReportNode
     │            │
     │            └── REJECT
     │                  ├── 分类驳回原因（5类：事实错误/逻辑矛盾/个性化不足/医学错误/不完整）
     │                  ├── 生成修正指引
     │                  ├── 权重衰减（×0.5）
     │                  └── reflection_count++
     │                        │
     │                  ┌─────┴─────┐
     │                  │ count < 3 │ count >= 3
     │                  │  重试     │ 强制输出
     └──────────────────┘           └──→ ReportNode
```
