# AI 应用中的 Planner 详解（含 CoT 深度扩展）

## 1. 什么是 Planner？

在 AI 应用里，**Planner（规划器）** 就像一个“总指挥”或“项目经理”。它的核心作用是把一个复杂的大任务，自动拆解成一系列更小、更可执行的步骤，并安排好这些步骤的顺序和依赖关系。

- **核心任务**：将用户模糊、复杂的指令（如“规划一次云南七日游”）分解成具体的子任务（查机票、搜酒店、定路线……），并由执行模块调用工具完成。
- **为何重要**：大模型不擅长处理需要多步推理和操作的复杂任务，容易出错。Planner 提供了结构化的“思路”，大大提升任务成功率。
- **关键能力**：任务分解、根据前一步结果动态调整计划（自适应）、处理多步骤间的逻辑依赖。
- **典型应用场景**：  
  - AutoGPT、BabyAGI 等自主智能体  
  - RAG 应用（规划先去哪找数据、再搜什么网页）  
  - 企业自动化（“分析客户情绪”拆成拉数据、画像、调模型等步骤）

> 简单说，Planner 负责“想”和“规划”，而执行模块和工具负责“做”。没有它，AI 只能做单次问答；有了它，AI 才能像真正的智能体一样完成复杂任务。

## 2. 常见 Planner 实现方案对比

| 方案 | 思考方式 | 是否与外部交互 | 灵活性 | 成本与效率 | 典型适用场景 |
|------|----------|----------------|--------|------------|----------------|
| **CoT** (Chain of Thought) | 线性推理，不行动 | 否 | 低 | 低 | 数学题、逻辑推理、常识问答 |
| **ReAct** (Reason+Act) | 交替思考与行动 | 是 | 高 | 高（易循环、token多） | 搜索引擎问答、网页浏览、API操作 |
| **Plan-and-Solve** | 先完整计划，再逐步执行 | 是 | 中低 | 中 | 结构化自动化流程、报告生成 |
| **ToT** (Tree of Thoughts) | 树状探索与回溯 | 可结合 | 非常高 | 极高 | 复杂谜题、策略游戏、深度搜索 |

> *注：Plan-and-Solve 在执行步骤时可能调用工具，但计划本身不动态调整。*

### 2.1 CoT —— “自言自语，逐步推理”

- **工作流**：用户提问 → AI 生成隐含计划的推理文本 → AI 直接给出最终答案（不调用外部工具）
- **优点**：简单有效，提升复杂推理准确性，无需额外工具
- **缺点**：无法从外部获取新信息，容易基于错误前提继续推理

### 2.2 ReAct —— “边想边做，动态调整”

- **工作流**：思考（Thought） → 行动（Act，如搜索） → 观察（Observation） → 思考…… → 最终答案
- **优点**：灵活，能应对不确定性，可从环境反馈修正错误
- **缺点**：易陷入无限循环；token 消耗多；可能需要很多试错步骤

### 2.3 Plan-and-Solve —— “先想清楚，再一步一步做”

- **工作流**：接收任务 → 生成详细步骤列表 → 逐步执行（可能调用工具） → 汇总答案
- **优点**：计划清晰，可解释性强，执行有序
- **缺点**：缺乏灵活性，计划第一步错则全错，无法动态调整

### 2.4 ToT —— “探索多条路径，择优而行”

- **工作流**：每一步生成多个思考分支 → 形成思维树 → 通过评估选择最有希望的分支继续探索
- **优点**：能解决需要大量探索和全局规划的高难度问题
- **缺点**：计算开销极大，需大量调用模型

### 2.5 如何选择？

| 任务特征 | 推荐方案 |
|----------|----------|
| 简单，不依赖外部信息 | CoT |
| 需要联网或操作工具，环境多变 | ReAct |
| 流程固定，步骤明确 | Plan-and-Solve |
| 非常难，需要深度探索 | ToT（算力允许时） |

> 实际应用中常混合使用：例如先用 Plan-and-Solve 生成高层计划，每个步骤内部用 ReAct 执行，兼顾结构清晰与执行灵活。

## 3. ReAct 的实现框架

ReAct 是一个设计范式，而非独立框架。**LangChain** 和 **LangGraph** 是主流实现工具。

### 3.1 LangChain vs LangGraph

| 特性 | LangChain | LangGraph |
|------|-----------|-----------|
| 编排方式 | 线性执行（如 `create_react_agent`） | 图结构（Graph），原生支持循环、分支、状态持久化 |
| 适用场景 | 快速原型验证、简单任务 | 生产级复杂应用，需要精细控制 |
| 状态管理 | 弱 | 强（内置状态持久化） |
| 推荐度 | 入门友好 | **首选**（尤其对生产环境） |

> 在 LangChain 生态中，对于需要稳定性和可控性的应用，已逐渐转向 LangGraph。

### 3.2 在 LangGraph 中实现 ReAct

#### 方案一：高阶 API（`create_react_agent`）—— 开箱即用

```python
from langgraph.prebuilt import create_react_agent
from langchain_anthropic import ChatAnthropic

# 定义工具
def search(query: str):
    return f"搜索结果: {query}"

# 初始化模型和代理
model = ChatAnthropic(model="claude-3-sonnet-20240229")
agent = create_react_agent(model, tools=[search])

# 调用
result = agent.invoke({"messages": [{"role": "user", "content": "查找今天天气"}]})
