大模型项目的开发并非“一口吃成胖子”的简单问答，而是一条环环相扣的精密链条。这条链条，就是我们常说的**工作流**。在深入具体步骤之前，我们先把“工作流”这个概念本身讲透，再展开两大主流项目流程，最后厘清经常被拿来比较的“Skill”和“MCP”究竟如何各司其职。

---

## 一、到底什么是“工作流”？

**工作流就是“为了稳定地完成一件事，所设定的一系列前后衔接、可复现的步骤”。** 它回答了三个核心问题：

- **先做什么，后做什么**（顺序与依赖）
- **每一步由谁来做、用什么工具做**（角色与工具）
- **上一步产出什么，下一步需要什么**（输入与输出）

你可以把它理解为三道生活中的“流水线”：  

- **菜谱**：准备食材 → 切菜 → 开火烹饪 → 装盘，顺序一乱，菜就可能糊掉。  
- **工厂装配线**：每个工位只负责一个固定环节，最终组装出完整产品。  
- **体检流程**：登记 → 抽血 → 内科 → 取报告，跳过抽血就直接内科，结果就不完整。

**一个标准的工作流，通常包含四个要素**：

| 要素                         | 含义                             | 大模型项目举例                             |
| ---------------------------- | -------------------------------- | ------------------------------------------ |
| **节点 (Steps)**             | 具体要执行的任务                 | 数据清洗、模型评测、部署上线               |
| **流转 (Flow)**              | 步骤间的先后顺序与依赖关系       | 必须先完成预训练，才能进行监督微调         |
| **输入/输出 (I/O)**          | 上一步的产出如何成为下一步的原料 | 清洗后的语料 → 训练框架的数据接口          |
| **角色/工具 (Actors/Tools)** | 由人完成，还是由系统自动调度     | 工程师搭建集群、自动化评测脚本、CI/CD 管道 |

### 为什么大模型项目如此强调工作流？

因为大模型开发是一个**高投入、强依赖、多角色**的系统工程。如果没有固定的工作流，极容易陷入：

- **混乱**：模型还没开始训练，显卡就先买了一大批闲置。
- **返工**：部署上线后，才发现训练数据含有隐私信息，需要重新清洗、重新训练。
- **不可复现**：效果好时不知道为什么好，效果崩了也无法精准定位问题环节。

因此，预先设计好工作流，本质上就是**把不确定性关进流程的笼子里**，用确定的过程来逼近高质量的结果。

## 二、大模型项目的两大主流工作流

根据项目目标与资源的不同，大模型工作流通常分为两大类。以下是两者侧重点的完整对比与详细拆解。

### 第一类：从零训练大模型（或垂直领域大规模继续训练）

**适合场景**：自研基础模型、行业大模型（如金融、医疗、法律）。  
**核心特征**：资源密集、工程繁重、周期长，追求掌握模型能力的“源头”。

**详细流程**：

1. **需求与可行性评估**  
   明确模型的能力边界、业务指标（如准确率、推理速度），估算算力、数据规模与合规红线。这一步的失误是最高成本的失误。

2. **数据采集与预处理**  
   从网页、书籍、代码、论文等多源渠道采集原始数据，经过严格的**去重、清洗、低质内容过滤、隐私脱敏**，构建高质量训练语料。这是“垃圾进、垃圾出”定律的起点。

3. **基座模型架构设计与训练框架搭建**  
   选定模型结构（Dense、MoE 等），配置分布式训练环境（GPU 集群、DeepSpeed/Megatron 等框架），并搭建训练监控、日志与断点续训系统。

4. **大规模预训练**  
   开展无监督/自监督训练，通过数据并行、流水线并行、张量并行等优化策略，持续监控 Loss 与训练稳定性，完成预设 Token 量的训练。这一阶段消耗了项目 80% 以上的时间和算力。

5. **监督微调 (SFT)**  
   构建高质量的指令数据集（多轮对话、任务提示），让模型学会遵循对话格式和执行具体任务，从“知识库”进化为“可用助手”。

6. **对齐训练 (RLHF/DPO)**  
   训练奖励模型，或直接使用偏好数据，通过人类反馈强化学习 (RLHF) 或直接偏好优化 (DPO) 等方法，让输出更符合人类价值观，降低有害、偏见内容。

7. **模型评测**  
   建立自动化评测基准（知识、推理、代码、安全等维度）与人工评测体系，进行红队攻击测试，验证各维度是否达到预设标准。

8. **推理优化与部署**  
   通过量化、剪枝、蒸馏等手段压缩模型，部署 vLLM、TensorRT-LLM 等高性能推理引擎，实现低延迟、高吞吐的在线服务。

9. **应用集成**  
   对接业务系统，支持检索增强生成 (RAG)、工具调用 (Agent) 等高级玩法，形成可落地的产品。

10. **监控与持续迭代**  
    收集线上反馈与 Badcase，驱动下一轮数据补充、微调甚至继续预训练，形成数据飞轮。

---

### 第二类：基于现有大模型的应用开发（微调/无微调）

**适合场景**：基于 GPT-4o、Qwen、Llama 等开放或闭源模型，快速构建面向业务的产品。  
**核心特征**：重数据闭环与工程落地，追求敏捷迭代，将模型能力封装为产品。

**详细流程**：

1. **场景定义与目标设定**  
   明确任务类型（客服、摘要、代码生成、数据分析等），设定清晰的成功指标（准确率、召回率、延迟、成本约束）。

2. **基座模型选型与 Prompt 工程**  
   横向对比候选模型（性能、成本、吞吐），编写系统 Prompt 与少样本示例，用少量标注数据快速验证初版效果上限。

3. **数据构建与微调准备**  
   收集真实业务对话、人工撰写，或通过模型蒸馏（让强模型生成）等方式构建高质量微调数据（SFT、偏好数据）。

4. **微调与实验**  
   执行参数高效微调（LoRA/QLoRA）或全参数微调，管理实验版本（模型、超参、数据组合），严谨对比基线与微调后的效果差异。

5. **效果评估**  
   建立业务专属的评测集（边界 case、对抗样本），结合自动指标（ROUGE/BLEU、G-Eval、LLM-as-Judge）与人工打分，确保质量通过验收。

6. **高级能力集成**  
   按需接入 RAG（解决知识截止和幻觉）、多模态识别、Agent 工具链（搜索、代码执行），处理复杂逻辑和实时信息需求。

7. **部署、监控与持续优化**  
   自部署或通过 API 上线，实时监控成本、延迟、错误率、用户反馈，用 Badcase 驱动 Prompt 调整、知识库更新或定期微调。

## 三、Skill 和 MCP：谁更好？—— 其实是“手”与“脑皮层”的协同

在了解了整体工作流之后，经常有人问：“Skill 和 MCP，到底该用哪个？” 答案很明确：**它们不是替代品，而是一对各司其职、高度协同的搭档**。

我们可以这样理解它们的分工：

- **MCP (Model Context Protocol)**：是 **“手和脚”**，让模型能够**连接和操作**外部世界（查数据、调 API、操作文件）。
- **Skill (技能包)**：是 **“大脑皮层”**，为模型注入**专业流程和经验**，教它“怎么想、怎么做”才更规范。

### 核心定位对比

| 维度         | MCP                                  | Skill                                        |
| ------------ | ------------------------------------ | -------------------------------------------- |
| **本质**     | 打通外部工具和数据的标准化“万能接口” | 将专业知识、流程封装成可遵循的“SOP 操作手册” |
| **核心问题** | “我能调用什么能力？”                 | “我应该如何规范地执行？”                     |
| **知识类型** | 工具性知识（Tool Knowledge）         | 流程性知识（Procedural Knowledge）           |
| **载体形式** | MCP 服务器（JSON‑RPC 协议）          | 基于 Markdown 的文件夹，渐进式披露           |
| **加载方式** | 启动时注册工具列表                   | 先读元数据，再按需加载详情，极度节省上下文   |
| **核心优势** | 标准化集成、安全治理、跨模型复用     | 执行稳定、成果一致、低成本固化专业知识       |
| **典型场景** | 查天气、调 API、操作数据库、发邮件   | 写周报、代码审查、按品牌规范输出文案         |

### 协同实战案例

想象一个场景：“查询最新销售数据，生成一份符合公司规范的 PPT 简报”。工作流如下：

1. **Skill 规划** — AI 加载 `Quarterly_Report` Skill，该手册规定了必须包含销售额、增长率等核心指标，并指向公司 PPT 模板。  
2. **MCP 执行** — Skill 调度 `Database MCP` 服务器去查询过去一个季度的数据。  
3. **Skill 规范** — 接到数据后，Skill 强制要求图表必须使用指定的品牌配色，不得随意发挥。  
4. **MCP + Skill 生成** — `PPT Skill` 最终调用 `Python-pptx MCP` 工具生成完全合规的汇报文件。

**抽象公式**：  
> **通用大模型（大脑） + MCP（手脚与工具） + Skills（专业经验与流程） = 高质量、可落地的 AI 应用**

### 实战选型指南

- **只需固化一套规范**（如代码审查、文档生成） → **选 Skill**：轻量高效，成本极低，一个文件夹即封装全部经验。  
- **只需连接外部世界**（如查数据库、调天气 API） → **选 MCP**：这是安全、标准连接外部工具的唯一解。  
- **复杂生产力场景**（如上述简报生成、自动化运维处理） → **Skill + MCP 结合使用**，这是打造强健 AI 工作流的理想范式。

---

无论是从零训练大模型的百亿参数炼成，还是基于现有模型的敏捷应用开发，**清晰的工作流都是项目成功的基石**。而在应用层，理解 Skill 与 MCP 的“流程 + 工具”协同模式，则能让你的 AI 不只“聪明”，更“靠谱、可控”。你可以根据自身资源与目标，选取对应的流程作为蓝图，再按实际情况大胆裁剪、迭代。
