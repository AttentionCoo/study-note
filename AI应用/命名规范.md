### 🧑‍💻 编程语言命名规范 (Java 与 Python)

#### **Java 命名规范**

| 元素 | 命名规则 (Case) | 核心要求与示例 |
| :--- | :--- | :--- |
| **包 (Package)** | 全小写 (lowercase) | 使用点`.`分隔，基于组织的反向域名（域名反转）。<br>*例: `com.alibaba.druid`* |
| **类/接口 (Class/Interface)** | 大驼峰 (UpperCamelCase) | 名词或名词短语，首字母大写。接口常用形容词（如 `Runnable`）。<br>*例: `HTMLEditorKit`, `UserDTO`* |
| **方法 (Method)** | 小驼峰 (lowerCamelCase) | 动词或动词短语，首字母小写。<br>*例: `getUserById()`, `deposit()`* |
| **变量 (Variable)** | 小驼峰 (lowerCamelCase) | 名词或名词短语，首字母小写，简短且有意义。<br>*例: `accountNumber`, `userName`* |
| **常量 (Constant)** | 全大写 + 下划线 (UPPER_SNAKE_CASE) | `static final` 修饰，用下划线分隔单词，语义完整。<br>*例: `MAX_STOCK_COUNT`* |
| **枚举 (Enum)** | 类型: 大驼峰<br>值: 全大写+下划线 | 类与常量命名规则的结合。<br>*例: `enum DayOfWeek { SUNDAY, MONDAY }`* |
| **泛型 (Generic)** | 单个大写字母 | 通常用 `T` (Type), `E` (Element), `K` (Key), `V` (Value) 等。<br>*例: `public class Box<T>`* |

**🧐 进阶补充与硬性规则**
*   **特殊类命名模式**：
    *   **抽象类**：推荐以 `Abstract` 或 `Base` 开头，如 `AbstractUserService`。
    *   **异常类**：必须以 `Exception` 结尾。
    *   **测试类**：以被测试类名开头，`Test` 结尾，如 `UserServiceTest`。
    *   **数据对象**：常用后缀有 `DO` (Data Object), `DTO` (Data Transfer Object), `VO` (Value Object), `DAO` (Data Access Object) 等。
*   **命名硬性规则**：
    *   严禁使用拼音或中英文混合命名。
    *   禁止以下划线(`_`)或美元符号(`$`)开头/结尾。
    *   禁止使用无意义缩写（如 `cal` 代替 `calculate`），但允许约定俗成的缩写（如 `DAO`, `HTML`）。

#### **Python 命名规范 (PEP 8)**

| 元素 | 命名规则 (Case) | 核心要求与示例 |
| :--- | :--- | :--- |
| **包/模块 (Package/Module)** | 全小写 + 下划线 (snake_case) | 简短、全小写，可用下划线提高可读性。<br>*例: `my_package`, `utils`* |
| **类 (Class)** | 大驼峰 (CapWords/CamelCase) | 单词首字母大写，无分隔符。<br>*例: `MyClass`, `DataAnalyzer`* |
| **异常 (Exception)** | 大驼峰 (CapWords) | 与类命名规则相同，通常以 `Error` 结尾。<br>*例: `ValueError`* |
| **函数/方法 (Function/Method)** | 全小写 + 下划线 (snake_case) | 动词或动词短语。<br>*例: `get_student_name()`, `calculate_total`* |
| **变量/属性 (Variable/Attribute)** | 全小写 + 下划线 (snake_case) | 名词或名词短语。<br>*例: `user_id`, `total_count`* |
| **常量 (Constant)** | 全大写 + 下划线 (UPPER_SNAKE_CASE) | 通常在模块级别定义。<br>*例: `MAX_OVERFLOW`, `PI`* |
| **私有成员 (Private)** | 单下划线开头 (`_leading_underscore`) | 弱“内部使用”约定，`from module import *` 不会导入。<br>*例: `_internal_method()`, `_private_var`* |
| **避免命名冲突** | 单下划线结尾 (`trailing_underscore_`) | 用于避免与Python关键字冲突。<br>*例: `class_`, `list_`* |
| **特殊方法 (Magic)** | 双下划线开头和结尾 (`__dunder__`) | Python保留，用户不应自定义此类名称。<br>*例: `__init__`, `__str__`* |

**💡 补充说明**
*   **私有属性“伪私有”机制**：双下划线前缀 (`__private_var`) 会触发Python的名称改编 (Name Mangling)，用于避免子类属性冲突，不应随意使用。
*   **类型变量**：如Java的泛型，PEP 484建议类型变量使用大驼峰命名，如 `T`, `AnyStr`。

### 🌐 GitHub 协作规范

#### **仓库命名**

| 元素 | 命名规则 (Case) | 核心要求与示例 |
| :--- | :--- | :--- |
| **仓库名 (Repository)** | 小写短横线 (kebab-case) | 全小写，用短横线`-`分隔单词。避免使用大写、下划线或特殊字符。<br>*例: `user-management`, `deploy-cli`* |

**🧐 更多细节与最佳实践**
*   **简洁与描述性**：名称应准确反映项目内容，理想长度为2-4个单词。
*   **避免冗余词**：不要包含 `best`、`final`、`new` 等无意义形容词，或 `v1`、`dev` 等版本/环境标识。
*   **组织级约定**：大型组织常有内部前缀要求，如 `acme-`、`infra-`，应优先遵守。
*   **特定类型仓库**：某些生态有自己的规则，如Python包常用 `py-` 前缀 (`py-sema`)，GitHub Actions用 `-action` 后缀。
*   **开源示例**：
    *   `facebook/react`：简洁的主项目名。
    *   `pallets/flask`：Python Web框架，组织名简洁。

---

#### **分支命名**

| 元素 | 命名规则 (Case) | 核心要求与示例 |
| :--- | :--- | :--- |
| **分支 (Branch)** | 分类/描述 (小写短横线) | `<type>/<description>` 格式，描述使用小写短横线。<br>*例: `feature/user-login`, `bugfix/header-overflow`* |

**🧐 更多细节与最佳实践**
*   **常用分支类型前缀**：
    *   `feature/`：新功能开发。
    *   `bugfix/` 或 `fix/`：修复非紧急Bug。
    *   `hotfix/`：紧急修复生产环境问题。
    *   `release/`：准备新版本发布。
    *   `chore/`：杂项任务，如更新依赖。
    *   `docs/`：文档编写与修改。
    *   `refactor/`：代码重构。
*   **描述部分**：应简短清晰，如 `feature/user-authentication`。
*   **主流分支模型示例**：
    *   **GitHub Flow**：仅一个 `main` 分支，所有功能、修复均创建描述性分支（如 `feature/new-button`），通过Pull Request合并。
    *   **Git Flow**：除 `main` 外，还包含 `develop`（开发主线）、`feature/*`、`release/*`、`hotfix/*` 等。

---

#### **提交信息**

| 元素 | 命名规则 | 核心要求与示例 |
| :--- | :--- | :--- |
| **提交信息 (Commit Message)** | 约定式提交 (Conventional Commits) | `<type>(<scope>): <subject>` 格式。主题用祈使句，首字母小写。<br>*例: `feat(auth): add login support`* |

**🧐 更多细节与最佳实践**
*   **常用类型 (`type`)**：`feat` (新功能), `fix` (修复Bug), `docs` (文档), `style` (格式), `refactor`, `test`, `chore` 等。
*   **范围 (`scope`)**：可选的，表示影响范围，如 `feat(parser): add ability to parse arrays`。
*   **主题 (`subject`)**：
    *   如今时态祈使句。
    *   首字母不大写，结尾不加句号。
    *   简短精炼，建议不超过50个字符。
*   **正文 (`body`, 可选)**：详细描述，解释"为什么"做这个改动。
*   **脚注 (`footer`, 可选)**：关联Issue (#123)，或注明破坏性变更 (`BREAKING CHANGE:`)。
*   **自动化价值**：遵循"约定式提交"能自动生成变更日志 (Changelog)，并支持语义化版本 (SemVer)。

---

#### **标签与发布版本**

| 元素 | 命名规则 | 核心要求与示例 |
| :--- | :--- | :--- |
| **标签 (Tag)** | 语义化版本 (Semantic Versioning) | `v<major>.<minor>.<patch>` 格式，如 `v1.0.0`。 |

*   **语义化版本 (SemVer)**：
    *   **主版本号 (MAJOR)**：不兼容的API修改。
    *   **次版本号 (MINOR)**：向下兼容的功能性新增。
    *   **修订号 (PATCH)**：向下兼容的问题修正。
*   **标签与发布**：通常，`v1.0.0` 标签对应 `1.0.0` 版本。可以在GitHub上基于标签创建正式的发布版本（Release），并附带详细的发行说明。

---

#### **Pull Request (PR)**

*   **命名**：标题清晰概括变更内容，与分支名或提交信息风格对应，如 `feat: Add user login functionality`。
*   **描述**：应详细解释动机、改动、测试步骤，并可用关键词 (如 `Closes #123`) 关联议题。
*   **最佳实践**：保持PR专注和较小规模；一个PR解决一个问题，便于审查和跟踪。

### 💎 总结

命名规范的核心目标是 **提高代码的可读性、可维护性以及团队协作效率**。一个好的名字本身就是最好的文档，能让其他开发者（包括未来的你）快速、准确地理解代码的意图。严格遵守这些"约定俗成"的规范，能显著降低沟通成本，提升软件的整体质量。

| 领域 | 核心特点总结 |
| :--- | :--- |
| **Java** | 严谨如大型企业，命名模式精巧且严格区分大小写，以精确表达含义。 |
| **Python** | 简洁优雅，有"全小写+下划线"与"大驼峰"的清晰分界，并有助于避免命名冲突。 |
| **GitHub** | 协作的枢纽，标准化命名和流程（如约定式提交）使自动化工具和全球团队协作成为可能。 |

### 📦 核心源码包 (`src/` 或项目同名包)

这是代码的核心，按功能模块组织为 Python 子包。

```
src/
├── __init__.py
├── data/                     # 数据处理
│   ├── __init__.py
│   ├── dataset.py            # 自定义 Dataset 类
│   ├── transforms.py         # 数据增强、预处理
│   └── loader.py             # DataLoader 构建
├── models/                   # 模型定义
│   ├── __init__.py
│   ├── backbone.py           # 骨干网络 (如ResNet)
│   ├── heads.py              # 任务头 (分类、检测)
│   └── model_builder.py      # 模型组装工厂
├── training/                 # 训练相关
│   ├── __init__.py
│   ├── trainer.py            # 训练循环 (一个Epoch的逻辑)
│   ├── optimizer.py          # 优化器与学习率调度
│   └── losses.py             # 自定义损失函数
├── evaluation/               # 评估
│   ├── __init__.py
│   └── metrics.py            # 准确率、mAP等指标计算
├── inference/                # 推理
│   ├── __init__.py
│   └── predictor.py          # 单张图片/批量预测pipeline
├── utils/                    # 通用工具
│   ├── __init__.py
│   ├── logger.py             # 日志配置 (TensorBoard, Wandb)
│   ├── checkpoint.py         # 模型保存与加载
│   └── visualize.py          # 结果可视化
└── config/                   # 配置解析 (如果用代码管理配置)
    ├── __init__.py
    └── config.py             # dataclass 或 OmegaConf 解析
```

#### 关键模块职责详解

- **`data/`**：封装所有数据相关操作，提供统一的 `Dataset` 接口。这样模型训练时只需关心数据加载器，不必接触原始文件格式。
- **`models/`**：纯模型定义，不含训练逻辑。使用工厂模式 (`model_builder.py`) 可根据配置名称动态创建模型。
- **`training/`**：核心训练循环，负责前向传播、反向传播、梯度更新和日志记录。`trainer.py` 应独立于具体模型和数据，通过参数注入复用。
- **`evaluation/`**：封装评估指标，可在训练后或单独脚本中调用，保持评估一致性。
- **`inference/`**：封装预处理 -> 模型加载 -> 推理 -> 后处理的完整流程，是连接模型与应用的桥梁。

### 🧪 实验管理 (`experiments/`)

用于组织所有实验产物，按实验名或日期划分子目录。

```
experiments/
├── exp_classification_baseline/
│   ├── logs/                 # TensorBoard 事件文件
│   ├── checkpoints/          # 模型权重 (best.pth, last.pth)
│   └── config.yaml           # 该次实验的完整配置备份
└── exp_segmentation_v2/
    └── ...
```

**最佳实践**：每次运行脚本时自动创建一个带时间戳的实验目录，并将配置文件、代码快照和所有输出保存进去，确保实验完全可追溯。

### 🌐 AI 应用服务结构 (以 FastAPI 为例)

当需要将模型部署为 API 时，可添加 `api/` 目录。

```
api/
├── main.py                   # FastAPI 应用入口，路由定义
├── dependencies.py           # 依赖注入 (如加载模型单例)
├── routers/                  # 分组路由
│   ├── predict.py            # /predict 端点
│   └── health.py             # 健康检查
├── schemas/                  # Pydantic 数据模型
│   └── requests.py           # 请求/响应结构定义
├── services/                 # 业务逻辑
│   └── model_service.py      # 调用 src/inference 并处理结果
└── middleware/               # 日志、限流等中间件
```

这种结构将 Web 框架与核心 AI 代码清晰分离，`api/` 只负责请求处理和调度，实际推理逻辑依然在 `src/inference` 中。

### 🛠️ 辅助脚本 (`scripts/`)

放置可直接运行的顶层脚本。

```
scripts/
├── train.py                  # 训练入口 (解析命令行参数)
├── evaluate.py               # 批量评估
├── predict.py                # 命令行推理工具
├── preprocess_data.py        # 一次性数据预处理
└── download_dataset.py       # 数据下载脚本
```

这些脚本通常非常简短，负责解析命令行参数、加载配置，然后调用 `src` 中的相应模块。

### ⚙️ 配置管理方式

推荐将配置与代码分离，主要有两种主流方式：

1.  **YAML 文件** (`configs/` 目录) ：使用 `OmegaConf` 或 `yaml` 库加载。易于修改，无需改动代码。
    ```yaml
    # configs/default.yaml
    model:
      name: resnet50
      num_classes: 10
    training:
      batch_size: 32
      epochs: 100
    ```

2.  **代码配置** (`src/config/`) ：使用 Python 的 `dataclass` 或 `attrs`。拥有 IDE 提示和类型检查优势，适合复杂配置。

### 🚀 项目启动与安装

采用 **`src` 布局** 是现代推荐做法，能避免导入混淆。项目必须被安装（开发模式）后才可运行：

```bash
# pyproject.toml 中配置后，在项目根目录执行
pip install -e .
```

这样，你在任何脚本中都可以用 `from src.models import ...` 进行绝对导入，结构清晰且不会出现模块找不到的错误。

**只需要记住这一个核心原则：让模型定义、训练流程和数据加载相互独立，通过配置将它们组装起来，你的项目就能从单次实验轻松成长为可部署的应用。**
