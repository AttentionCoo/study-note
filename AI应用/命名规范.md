# 项目命名与结构规范完全指南

本指南涵盖三个核心领域：
- **编程语言命名规范**（Python 与 Java）
- **GitHub 协作规范**（仓库、分支、提交信息等）
- **Python 深度学习/AI 项目目录结构**

所有约定均以提升可读性、可维护性和团队协作效率为目标。

---

## 一、编程语言命名规范

命名主要体现在两个层面：**项目/仓库/包结构** 和 **代码内部元素**。下面分别对 Python 和 Java 做详细说明。

### 1.1 Python 命名规范（遵循 PEP 8）

#### 核心规则表

| 命名对象                        | 规范                            | 示例                                       |
| :------------------------------ | :------------------------------ | :----------------------------------------- |
| **项目 / 仓库名**               | `kebab-case`（全小写 + 短横线） | `my-image-classifier`， `flask-api`        |
| **顶层包名**（`import` 的名称） | `snake_case`（全小写 + 下划线） | `deep_vision`， `data_utils`               |
| **模块 / 文件名**               | `snake_case`                    | `model_trainer.py`， `losses.py`           |
| **类**（含异常）                | `CapWords`（大驼峰）            | `ResNet50`， `InvalidInputError`           |
| **函数 / 方法**                 | `snake_case`                    | `load_checkpoint()`， `compute_accuracy()` |
| **变量 / 属性**                 | `snake_case`                    | `learning_rate`， `total_loss`             |
| **常量**                        | `UPPER_SNAKE_CASE`              | `DEFAULT_BATCH_SIZE`， `MAX_OVERFLOW`      |
| **私有 / 内部成员**             | 单下划线前缀 `_`                | `_internal_helper()`， `_hidden_dim`       |
| **避免命名冲突**                | 单下划线结尾 `_`                | `class_`， `list_`                         |
| **特殊方法（魔术方法）**        | 双下划线开头和结尾 `__`         | `__init__`， `__str__`                     |
| **类型变量**（泛型）            | `CapWords`，通常单字母          | `T`， `AnyStr`                             |

#### 补充说明
- **私有属性“伪私有”机制**：双下划线前缀（`__private_var`）会触发名称改编（Name Mangling），用于避免子类属性冲突，**非必要不使用**。
- **异常命名**：通常以 `Error` 结尾，如 `ValueError`。
- **类型变量**：PEP 484 建议使用大驼峰（如 `T`, `K`, `V`）。

**快速记忆：**  
项目目录用 **短横线**，代码里面全是 **下划线**，只有 **类** 用大驼峰。

### 1.2 Java 命名规范

#### 核心规则表

| **项目/仓库名** | 推荐 `kebab-case`（全小写+短横线） | `user-management`， `order-service` |

| 元素              | 命名规则                 | 核心要求与示例                                               |
| :---------------- | :----------------------- | :----------------------------------------------------------- |
| **包（Package）** | 全小写（lowercase）      | 点`.`分隔，基于反向域名。<br>*例: `com.alibaba.druid`*       |
| **类 / 接口**     | 大驼峰（UpperCamelCase） | 名词或名词短语，首字母大写。接口常用形容词。<br>*例: `HTMLEditorKit`, `UserDTO`, `Runnable`* |
| **方法**          | 小驼峰（lowerCamelCase） | 动词或动词短语，首字母小写。<br>*例: `getUserById()`, `deposit()`* |
| **变量 / 参数**   | 小驼峰（lowerCamelCase） | 名词或名词短语，简短且有意义。<br>*例: `accountNumber`, `maxRetryCount`* |
| **常量**          | 全大写 + 下划线          | `static final` 修饰。<br>*例: `MAX_STOCK_COUNT`*             |
| **枚举类型**      | 大驼峰                   | 与类相同。<br>*例: `enum DayOfWeek`*                         |
| **枚举值**        | 全大写 + 下划线          | 与常量相同。<br>*例: `DayOfWeek.MONDAY`*                     |
| **泛型**          | 单个大写字母             | `T` (Type), `E` (Element), `K` (Key), `V` (Value)。<br>*例: `public class Box<T>`* |

#### 进阶补充与硬性规则
- **特殊类命名模式**：
  - 抽象类：推荐以 `Abstract` 或 `Base` 开头，如 `AbstractUserService`。
  - 异常类：必须以 `Exception` 结尾。
  - 测试类：以被测试类名开头，`Test` 结尾，如 `UserServiceTest`。
  - 数据对象：常用后缀 `DO`、`DTO`、`VO`、`DAO` 等。
- **硬性规则**：
  - 严禁拼音或中英文混合命名。
  - 禁止以下划线或美元符号开头/结尾。
  - 禁止无意义缩写（如 `cal` 代替 `calculate`），允许约定俗成缩写（`DAO`、`HTML`）。

**快速记忆：**  
包名 **全小写**，类名 **大驼峰**，方法/变量 **小驼峰**，常量 **全大写下划线**。

### 1.3 Python 与 Java 直观对比

| 层级        | Python                      | Java                |
| :---------- | :-------------------------- | :------------------ |
| 项目 / 仓库 | `my-project`                | `my-project`        |
| 包 / 模块   | `my_package` / `trainer.py` | `com.example.myapp` |
| 类          | `DataProcessor`             | `DataProcessor`     |
| 方法 / 函数 | `process_data()`            | `processData()`     |
| 变量        | `total_count`               | `totalCount`        |
| 常量        | `MAX_SIZE`                  | `MAX_SIZE`          |

> 核心差别在于方法/变量风格：Python 用 `snake_case`，Java 用 `lowerCamelCase`；包/模块风格完全不同。

## 二、GitHub 协作规范

统一的 Git 协作规范能显著提升团队开发效率，并为自动化工具（如变更日志生成、语义化版本）提供基础。

### 2.1 仓库命名

| 元素                     | 规则                            | 示例                             |
| :----------------------- | :------------------------------ | :------------------------------- |
| **仓库名（Repository）** | `kebab-case`（全小写 + 短横线） | `user-management`， `deploy-cli` |

#### 最佳实践
- 简洁且描述性强，2–4 个单词为佳。
- 避免使用 `best`、`final`、`new` 等无意义形容词，以及 `v1`、`dev` 等版本/环境标识。
- 组织内部可以约定前缀，如 `acme-`、`infra-`；生态项目常有特殊习惯（如 Python 包用 `py-` 前缀，GitHub Actions 用 `-action` 后缀）。
- 开源示例：`facebook/react`、`pallets/flask`。

### 2.2 分支命名

| 规则                   | 格式                | 示例                                            |
| :--------------------- | :------------------ | :---------------------------------------------- |
| `<type>/<description>` | 描述用 `kebab-case` | `feature/user-login`， `bugfix/header-overflow` |

#### 常用分支类型
- `feature/`：新功能开发。
- `bugfix/` 或 `fix/`：修复非紧急 Bug。
- `hotfix/`：紧急修复生产环境问题。
- `release/`：准备新版本发布。
- `chore/`：杂项任务（如更新依赖）。
- `docs/`：文档编写与修改。
- `refactor/`：代码重构。

#### 主流分支模型
- **GitHub Flow**：仅 `main` 分支，功能/修复均创建描述性分支，通过 PR 合并。
- **Git Flow**：除 `main` 外还有 `develop`，并配合 `feature/*`、`release/*`、`hotfix/*`。

### 2.3 提交信息（Commit Message）

采用 **约定式提交（Conventional Commits）**。

| 规则                         | 格式                     | 示例                            |
| :--------------------------- | :----------------------- | :------------------------------ |
| `<type>(<scope>): <subject>` | 主题用祈使句，首字母小写 | `feat(auth): add login support` |

#### 常用类型（type）
- `feat`：新功能。
- `fix`：修复 Bug。
- `docs`：文档。
- `style`：格式（不影响代码逻辑）。
- `refactor`：重构。
- `test`：测试。
- `chore`：构建、依赖等杂项。

#### 编写细节
- **范围（scope）**：可选，表示影响范围，如 `feat(parser): add ability to parse arrays`。
- **主题（subject）**：现在时祈使句，首字母不大写，结尾不加句号，建议不超过 50 个字符。
- **正文（body）**：可选，解释“为什么”做这个改动。
- **脚注（footer）**：关联 Issue（如 `Closes #123`），或注明破坏性变更（`BREAKING CHANGE:`）。

遵循该规范可自动生成变更日志（Changelog）并支持语义化版本。

### 2.4 标签与发布版本

| 规则                 | 格式                       | 示例     |
| :------------------- | :------------------------- | :------- |
| 语义化版本（SemVer） | `v<MAJOR>.<MINOR>.<PATCH>` | `v1.0.0` |

- **主版本号（MAJOR）**：不兼容的 API 修改。
- **次版本号（MINOR）**：向下兼容的功能新增。
- **修订号（PATCH）**：向下兼容的问题修正。

通常基于标签创建 GitHub Release，并附带详细发行说明。

### 2.5 Pull Request（PR）

- **标题**：清晰概括变更内容，与分支或提交风格一致，如 `feat: Add user login functionality`。
- **描述**：详细解释动机、改动、测试步骤，可用关键词关联 Issue（`Closes #123`）。
- **最佳实践**：保持 PR 专注且小规模，一个 PR 解决一个问题，便于审查和追踪。

## 三、Python 深度学习 / AI 项目目录结构

规范的目录结构能够保证项目 **可复现、可维护、可部署**，兼顾研究实验与应用落地。

### 3.1 推荐根目录结构

```
my-dl-project/
├── README.md                 # 项目说明、环境配置、快速开始
├── LICENSE                   # 开源许可证
├── .gitignore                # Git 忽略文件（数据集、模型权重、虚拟环境）
├── requirements.txt          # 依赖库列表（或 environment.yml）
├── pyproject.toml            # 现代 Python 项目配置（取代 setup.py）
├── Makefile                  # 常用命令快捷方式（train, test, clean）
├── Dockerfile                # 容器化部署
├── configs/                  # 配置文件（YAML / JSON）
├── data/                     # 原始数据 & 处理后数据（通常被 .gitignore）
├── notebooks/                # 探索性数据分析与可视化
├── src/                      # 核心源码（或直接用项目名命名）
├── scripts/                  # 独立的训练、评估、预处理脚本
├── experiments/              # 实验记录、模型检查点、日志
├── tests/                    # 单元测试与集成测试
├── docs/                     # 详细文档
└── api/                      # （可选）Web 服务接口
```

### 3.2 核心源码包（`src/` 或项目同名包）

代码按功能模块组织为子包，保持高内聚低耦合。

```
src/
├── __init__.py
├── data/                     # 数据处理
│   ├── __init__.py
│   ├── dataset.py            # 自定义 Dataset 类
│   ├── transforms.py         # 数据增强、预处理
│   └── loader.py             # DataLoader 构建
├── models/                   # 模型定义
│   ├── __init__.py
│   ├── backbone.py           # 骨干网络（如 ResNet）
│   ├── heads.py              # 任务头（分类、检测）
│   └── model_builder.py      # 模型组装工厂
├── training/                 # 训练相关
│   ├── __init__.py
│   ├── trainer.py            # 训练循环（一个 Epoch 的逻辑）
│   ├── optimizer.py          # 优化器与学习率调度
│   └── losses.py             # 自定义损失函数
├── evaluation/               # 评估
│   ├── __init__.py
│   └── metrics.py            # 准确率、mAP 等指标计算
├── inference/                # 推理
│   ├── __init__.py
│   └── predictor.py          # 单张图片 / 批量预测 pipeline
├── utils/                    # 通用工具
│   ├── __init__.py
│   ├── logger.py             # 日志配置（TensorBoard, Wandb）
│   ├── checkpoint.py         # 模型保存与加载
│   └── visualize.py          # 结果可视化
└── config/                   # 配置解析（如果用代码管理配置）
    ├── __init__.py
    └── config.py             # dataclass 或 OmegaConf 解析
```

#### 各模块职责详解
- **`data/`**：封装所有数据操作，提供统一的 `Dataset` 接口，使训练时只需关心数据加载器，不接触原始文件格式。
- **`models/`**：纯模型定义，不含训练逻辑。`model_builder.py` 实现工厂模式，根据配置名称动态创建模型。
- **`training/`**：核心训练循环，负责前向传播、反向传播、梯度更新与日志记录。`trainer.py` 应独立于具体模型和数据，通过参数注入复用。
- **`evaluation/`**：封装评估指标，可在训练后或单独脚本中调用，保持评估一致性。
- **`inference/`**：封装预处理 → 模型加载 → 推理 → 后处理的完整流程，是连接模型与应用的关键桥梁。
- **`utils/`**：通用工具函数，如日志、可视化、检查点管理，供其他模块共享。
- **`config/`**（可选）：当使用 Python 代码管理配置时，集中存放配置类。

### 3.3 实验管理（`experiments/`）

按实验名称或日期组织所有实验产物，确保完全可追溯。

```
experiments/
├── exp_classification_baseline/
│   ├── logs/                 # TensorBoard 事件文件
│   ├── checkpoints/          # 模型权重（best.pth, last.pth）
│   └── config.yaml           # 该次实验的完整配置备份
└── exp_segmentation_v2/
    └── ...
```

**最佳实践**：每次运行脚本时自动创建带时间戳的实验目录，并将配置文件、代码快照和所有输出保存进去。

### 3.4 API 服务结构（基于 FastAPI 示例）

当需要将模型部署为 Web API 时，可添加 `api/` 目录，与核心 AI 代码清晰分离。

```
api/
├── main.py                   # FastAPI 应用入口，路由定义
├── dependencies.py           # 依赖注入（如加载模型单例）
├── routers/                  # 分组路由
│   ├── predict.py            # /predict 端点
│   └── health.py             # 健康检查
├── schemas/                  # Pydantic 数据模型
│   └── requests.py           # 请求 / 响应结构定义
├── services/                 # 业务逻辑
│   └── model_service.py      # 调用 src/inference 并处理结果
└── middleware/               # 日志、限流等中间件
```

- `api/` 只负责请求处理和调度，实际推理逻辑仍在 `src/inference` 中。
- `dependencies.py` 管理模型单例，避免重复加载。
- `schemas/` 定义清晰的请求/响应结构，便于自动生成 API 文档。

### 3.5 辅助脚本（`scripts/`）

放置可直接运行的顶层脚本，通常非常简短，负责解析命令行参数、加载配置，然后调用 `src` 中的相应模块。

```
scripts/
├── train.py                  # 训练入口
├── evaluate.py               # 批量评估
├── predict.py                # 命令行推理工具
├── preprocess_data.py        # 一次性数据预处理
└── download_dataset.py       # 数据下载脚本
```

### 3.6 配置管理方式

推荐将配置与代码分离，主要有两种主流方式：

1. **YAML 文件（`configs/` 目录）**  
   使用 `OmegaConf` 或 `yaml` 库加载，易于修改，无需改动代码。
   ```yaml
   # configs/default.yaml
   model:
     name: resnet50
     num_classes: 10
   training:
     batch_size: 32
     epochs: 100
   ```

2. **代码配置（`src/config/`）**  
   使用 Python `dataclass` 或 `attrs`，拥有 IDE 提示和类型检查优势，适合复杂动态配置。

### 3.7 项目启动与安装

采用 **`src` 布局**（现代推荐做法），项目必须通过开发模式安装后方可运行，避免导入混淆。

```bash
# 在 pyproject.toml 中正确配置后，于项目根目录执行
pip install -e .
```

之后，在任何脚本中都可以使用绝对导入 `from src.models import ...`，结构清晰，模块查找无烦恼。

## 四、总结

| 领域            | 核心特点                                                     |
| :-------------- | :----------------------------------------------------------- |
| **Python 命名** | 简洁优雅：全小写 + 下划线，类用大驼峰，私有用单下划线。      |
| **Java 命名**   | 严谨规范：包全小写，类大驼峰，方法/变量小驼峰，常量全大写下划线。 |
| **GitHub 协作** | 标准化分支、约定式提交、语义化版本，使团队协作与自动化工具无缝衔接。 |
| **AI 项目结构** | 模块分离（数据、模型、训练、评估、推理），配置与代码解耦，实验可追溯，易于从研究走向部署。 |

**核心原则：让模型定义、训练流程和数据加载相互独立，通过配置将它们组装起来。** 这样，你的项目就能从单次实验轻松成长为可部署的应用。
